# Блок 3: переосмысление гипотез и backlog (kanban + timeline)

**Дата:** 2026-05-08
**Ветка:** `extra-content-supplements`
**Контекст:** отборочное «Мини-CEO Т-Банк» (май 2026), Блок 3 — план достижения 60% автоматизации.

## Проблема

Сейчас гипотезы по Блоку 3 разнесены по двум секциям:

- `3.5 Основные гипотезы` — 7 строк-таблицы. Внутри смешаны «то, что закладывается в путь к 60%» и «то, что я только проверяю в Q1, без планового вклада» (UX-эскалации, разбивка 9,6%).
- `3.6 Backlog-гипотезы` — 8 строк-таблицы. У каждой указано «кому отдать», но нет явного маркера «можно взять как самостоятельную задачу для доп. оценки на ревью» vs «только при появлении ресурса».

Читателю (PM-проверяющий на собеседовании) приходится мысленно делить таблицу 3.5 пополам и догадываться о статусе backlog-задач. Связь с roadmap (3.4) не выводится наружу.

## Цели

1. **Видна разница** «в плане» / «проверяю» / «в backlog» с первого взгляда — без чтения колонок.
2. **Связь с roadmap** явная: видно, в какую волну попадает каждая плановая гипотеза.
3. **В backlog** виден маркер «эта задача декомпозирована до самостоятельного артефакта — кандидат может взять её как доп. работу на ревью».
4. Подача **интуитивная и не перегруженная** — без длинных таблиц, без дублирования между 3.5 и 3.6.

## Не-цели

- Не меняем суть гипотез, цифры вкладов, периоды или распределение ресурсов — это всё уже согласованный контент.
- Не меняем содержание других разделов Блока 3 (3.1–3.4, 3.7–3.10) кроме нумерации.
- Не делаем drag-and-drop kanban или live-фильтры — это перегруз для печатной презентации.

## Решение одним абзацем

Объединяю 3.5 и 3.6 в одну новую секцию `3.5 Что в плане, что проверяю, что в backlog`. Внутри — сегмент-контрол с двумя режимами просмотра (`Kanban` по умолчанию, `Timeline` опционально). Все **19 элементов** объединяются в единый набор карточек с тремя статусами (`plan` / `validate` / `backlog`) и переключаются между двумя видами без перезагрузки. Старая 3.6 удаляется, нумерация 3.7–3.10 сдвигается на 1.

## Источники данных и логика разбивки

В исходных секциях гипотезы и инициативы немного отличаются: 3.5 — это «гипотезы», 3.4 — «инициативы roadmap». В новом дизайне:

- **`В плане`** = строки из таблицы 3.4 «Roadmap, вклад и обоснование» (то, под что заложено время команды и есть период). 9 шт.
- **`Проверяю`** = инициативы без планового вклада или гипотезы из 3.5, требующие Q1 discovery до решения «расширять или нет». 2 шт.
- **`Backlog`** = все 8 строк из 3.6.

Гипотеза «UX-спровоцированные эскалации» (3.5) и инициатива «UX-выход из сценария» (3.4) **объединяются в одну карточку** статуса `plan`: время команды заложено (М1–М2), а разбор логов — её часть. Не дублируется как отдельная карточка `validate`.

## Архитектура секции

### Структура HTML

```html
<section id="b3-hypotheses">
  <div class="section-label">3.5 / Что в плане, что проверяю, что в backlog</div>
  <h2>
    <span>Гипотезы и backlog</span>
    <div class="view-switcher" role="tablist">
      <button data-view="kanban" class="is-active">Kanban</button>
      <button data-view="timeline">Timeline</button>
    </div>
  </h2>
  <div class="hyp-legend">…3 строки легенды…</div>

  <div class="hyp-board" data-view="kanban">
    <article class="hyp-card" data-status="plan" data-start="1" data-end="3">…</article>
    …19 карточек…
  </div>
</section>
```

Один общий dataset из 19 `.hyp-card`. Layout определяется атрибутом `data-view` на контейнере. CSS переключает grid-структуру; JavaScript только обновляет `data-view` и сохраняет в localStorage.

### Сегмент-контрол

- Две кнопки в одной обёртке `<div class="view-switcher">`, активная подсвечена жёлтым `var(--yellow)`, неактивная — outline.
- Размещение: справа от `<h2>`, выровнено по правому краю, отступ слева — auto.
- Поведение: клик меняет атрибут `data-view` на `.hyp-board`, обновляет `aria-selected`, сохраняет в `localStorage('miniceo:hyp-view')`.
- Дефолт при первой загрузке: `kanban`.
- На <720 px кнопка `Timeline` скрывается, режим принудительно `kanban` (см. адаптив).

## Режим A: Kanban

CSS Grid с тремя колонками равной ширины (`1fr 1fr 1fr`), gap 16 px. Заголовок каждой колонки — крупный (font-display 14 px, uppercase) с цветным маркером и счётчиком: `🟢 В плане · 9`, `🟡 Проверяю · 2`, `⚪ Backlog · 8`. Под заголовком — короткое 1-строчное описание категории.

Внутри колонок карточки выстраиваются вертикальным flex-стеком с gap 10 px, выровнены по верху. CSS-селекторы `[data-status="plan"]`, `[data-status="validate"]`, `[data-status="backlog"]` распределяют карточки по колонкам через `order` или `display:none`/`grid-area`.

## Режим B: Timeline

Layout — две зоны:

**1. Lanes-зона** (для `plan` и `validate`). Полоса от М1 до М24, разметка по месяцам с засечками на М1, М3, М6, М9, М12, М18, М24. Сверху — 4 цветные полосы волн (Волна 1 М1–М3 серый; Волна 2 М4–М9 жёлтый; Волна 3 М7–М12 жёлтый; Волна 4 М13–М24 жёлтый), позиционируются абсолютно через `left/width`, рассчитанные из CSS variables `--start`/`--end`.

Под волнами — чекпоинты автоматизации из 3.4 (`М3: 31–32%`, `М6: 34–36%`, `М9: 37–41%`, `М12: 40–46%`, `М18: 49–55%`, `М24: 60%`) как маленькие кружки с подписью.

Под чекпоинтами — карточки `plan` и `validate` в lanes (горизонтальных дорожках). Каждая инициатива в своей дорожке — пересечений нет. Позиция карточки: `left: calc(var(--start) / 24 * 100%); width: calc((var(--end) - var(--start)) / 24 * 100%)`.

**2. Backlog-зона** (фиксированно под lanes). Серая зона с лейблом `⚪ Backlog · без позиции на timeline · 8`. Внутри — те же карточки `data-status="backlog"`, разложенные в одну строку с горизонтальным переносом (flex-wrap). Это визуально подтверждает: «у backlog нет обещания по сроку».

## Печать (PDF / Ctrl+P)

`@media print` принудительно показывает Kanban-вид, скрывает Timeline. Сегмент-контрол скрывается. Карточки печатаются без теней. Backlog-колонка может уйти на следующую страницу через `break-before: auto`. Короткая PDF-версия (`IarochkinMiniCEO.pdf`) — отдельный документ, остаётся как был.

## JavaScript-логика

Минимальная — один обработчик `setupHypothesesView()` в `assets/script.js`:

```
1. Читает localStorage('miniceo:hyp-view'), если нет — 'kanban'
2. Применяет data-view на .hyp-board, обновляет aria-selected кнопок
3. Слушатель клика на .view-switcher button:
   - устанавливает data-view
   - обновляет aria-selected
   - сохраняет в localStorage
4. Слушатель resize: при <720px и data-view="timeline" принудительно переключает на kanban
```

Подключается в bootstrap-секции script.js рядом с `setupTheme()`, `setupRoadmap()`, etc.
